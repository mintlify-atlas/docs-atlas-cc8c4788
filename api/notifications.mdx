The Notifications API exposes a read-only view of MindFlow's automated reminder system. The backend runs a cron job every hour that scans for tasks with deadlines within the next 24 hours, checks for pending micro-objectives, and dispatches reminders subject to session-suppression and frequency-cap rules. This endpoint lets students (and frontend dashboards) inspect the full dispatch log — delivery statuses, attempt counts, and timestamps — without triggering any new dispatch themselves.

<Note>
  This is a **read-only** endpoint. Notifications are triggered exclusively by the hourly cron job running inside the backend service. There is no API call that initiates a notification dispatch.
</Note>

---

## GET /api/v1/notifications

Returns the full notification log for the authenticated student. Each entry in the log corresponds to a single notification event for a specific task, including its current delivery status, the number of dispatch attempts made, and the UTC timestamp of the most recent attempt.

**Authentication:** JWT Bearer

**Request Body:** None

### Response — 200 OK

<ResponseField name="data" type="array">
  Array of `NotificationLog` records for the authenticated student. Ordered by `dispatchedAtUtc` descending (most recent first).
</ResponseField>

<ResponseField name="error" type="null">
  Always `null` on success.
</ResponseField>

<ResponseField name="status" type="integer">
  `200`
</ResponseField>

Each item in the `data` array contains:

<ResponseField name="data[].id" type="string">
  UUID of the notification log entry.
</ResponseField>

<ResponseField name="data[].taskId" type="string">
  UUID of the task that triggered this notification.
</ResponseField>

<ResponseField name="data[].status" type="string">
  Delivery status of this notification. One of `pending`, `sent`, or `failed`. See the status reference table below.
</ResponseField>

<ResponseField name="data[].attemptCount" type="integer">
  The number of dispatch attempts made for this notification entry. Ranges from `0` (suppressed before any attempt) to `3` (maximum attempts exhausted).
</ResponseField>

<ResponseField name="data[].dispatchedAtUtc" type="string">
  ISO 8601 UTC timestamp of the most recent dispatch attempt or suppression event.
</ResponseField>

### Status Values

| Status | Meaning |
|--------|---------|
| `pending` | A dispatch attempt was made but has not yet succeeded. The cron job will retry on its next run, up to the 3-attempt maximum. |
| `sent` | The notification was delivered successfully. |
| `failed` | All 3 consecutive delivery attempts failed. No further retries will be made for this log entry. |

<Warning>
  A `failed` status is final. Once a `NotificationLog` entry reaches `attemptCount = 3` with no successful delivery, it is permanently marked `failed` and the cron job will not retry it. A new log entry may be created in a future cron run if the task's deadline is still within the 24-hour window and the student has not yet reached the daily frequency cap.
</Warning>

### Dispatch Rules (Cron Job Behaviour)

The following rules govern when the cron job creates or updates notification log entries. Understanding them helps interpret the log data returned by this endpoint.

**Eligibility:** A task is eligible for a notification if `is_deleted = false`, its `deadline` falls within the next 24 hours, and it has at least one micro-objective where `is_completed = false` and `is_audit_only = false`.

**Session suppression:** If the student has an active EMA session (`is_active = true`) at the time the cron job runs, the notification is suppressed and logged with `status = 'suppressed'` and `attemptCount = 0`. Suppressed entries do not appear in this endpoint's response (they are stored in the database for internal auditing but filtered from the student-facing log).

**Frequency cap:** The cron job tracks how many `sent` notifications a student has received within the current 24-hour rolling window. Once that count reaches **3**, no further notifications are created or dispatched for that student until the window resets — not even for different tasks.

**Retry logic:** For a given student–task pair, if a log entry exists with `status = 'pending'` or `status = 'failed'` and `attemptCount < 3`, the next cron run increments the attempt count and tries again. If `attemptCount` reaches `3` without a successful delivery, the entry is set to `status = 'failed'` and no further retries occur.

<Note>
  Notifications are suppressed for any student who has an active EMA session (`is_active = true`). This prevents interrupting the fatigue-assessment flow with deadline reminders. Once the session ends, normal dispatch resumes on the next hourly cron run.
</Note>
